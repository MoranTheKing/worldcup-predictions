# Handoff - 2026-04-22

## Product state

By 2026-04-22, the app was in the middle of the social predictions phase:

- global auth wrapped the app
- `/game` served as the protected social predictions hub
- private leagues supported create, join, leave, remove member, and delete
- opponent view already hid scheduled matches and unresolved placeholders
- the UI had landed total hits, joker split, and premium locked-outright badges

## Important update from the 2026-04-23 audit

Several security issues were found in the social layer after this handoff:

- permissive SELECT RLS on social prediction tables
- missing DB-level kickoff locks
- pre-kickoff outright serialization leaks
- direct join-page invite-code enumeration
- open redirects in auth entry points
- remotely reachable dev tooling in non-production

Those fixes are now documented in:

- `SECURITY_AUDIT_2026-04-23.md`

And require these new migrations:

- `20260423000018_restore_social_prediction_privacy.sql`
- `20260423000019_enforce_prediction_lock_windows.sql`

## Additional product update from 2026-04-23

The auth flow no longer drops fresh users into the game anonymously:

- a dedicated `/onboarding` flow now runs after login/signup until setup is complete
- each player must choose a unique nickname before reaching `/game` or `/dashboard`
- pre-kickoff onboarding also captures winner and top-scorer picks
- avatar selection is optional and supports default built-in avatars plus the current Google avatar when present
- nickname uniqueness is enforced in the database by `20260423000020_enforce_unique_profile_handles.sql`
- provider-prefilled users still start on the profile step first so they can edit before moving on
- onboarding now also supports secure personal avatar uploads from phone/desktop
- once inside the app, nickname and avatar can still be edited from the profile card in the game shell
- uploaded personal avatars are not exposed from a public bucket; they are validated server-side and served through an authenticated app route
- stored arbitrary local avatar paths are no longer accepted, which closes the same-origin GET / stored-CSRF risk described in PR #15
- private uploaded-avatar routes are accepted only as relative `/api/profile/avatar/...` URLs, closing the PR #17 external-origin bypass
- post-kickoff signup still requires confirming a public profile nickname/avatar, but no longer asks for locked outright picks
- personal avatar uploads now show an immediate in-form preview before save
- onboarding and post-login profile editing now include avatar framing controls for x/y positioning and zoom
- real profile avatars in the UI can now be clicked for a larger lightbox-style preview without changing any game logic
- the avatar framing sliders now honor the real reachable crop area, including vertical movement under zoom
- the profile editor opens from the top and presents nickname editing before avatar controls on compact layouts
- onboarding now mirrors that ordering: the nickname field is first and focus no longer scrolls the page down
- optional Authenticator/TOTP setup is available after email signup and is enforced only for accounts with a verified TOTP factor

The predictions hub also received a UI cleanup on 2026-04-23:

- the outright-picks block on `/game/predictions` was simplified to reduce redundant copy
- save controls and current outright selections now sit in a tighter, more visual layout
- the editable state no longer repeats the same winner/top-scorer values twice on screen
- live match UI now relies on an explicit `match_phase` field, so minute 45 can still be live, `45+` stoppage time can be shown, and halftime is set intentionally by DB/API state

## Browser QA and auth follow-up from 2026-04-24

The latest local browser pass covered the signup/onboarding/dashboard path:

- duplicate or already-confirmed signup attempts no longer advance to the OTP entry screen as if a new email was sent
- the resend message now explains that resend only helps pending verification addresses and that existing users should log in
- a fresh signup completed the full onboarding wizard, saved a nickname plus opening outright picks, and landed inside `/dashboard/matches`
- `/dashboard/matches` rendered the 104-match list after onboarding
- `/dashboard/tournament` now reads public tournament data server-side with a narrowed admin projection, so RLS no longer blanks the public tournament page
- Next 16 console warnings from dashboard flag images were addressed by giving flag `Image` components explicit CSS dimensions
- the root `<html>` now declares `data-scroll-behavior="smooth"` to match the global smooth-scroll CSS expected by Next 16
- signup now includes a live strong-password panel and blocks weak passwords before calling Supabase; Supabase Password Security still needs matching dashboard enforcement
- signup now also requires password confirmation before the OTP request is sent
- Google OAuth will continue showing the Supabase project domain until a paid Supabase Custom Domain such as `auth.moran65.com` is activated and added to the Google callback URLs
- `/dashboard/tournament` no longer uses the Supabase service-role client; public tournament data is exposed through the narrow views `public_tournament_matches` and `public_tournament_teams`
- signup account-enumeration handling is now neutral: the OTP screen does not reveal whether an email exists, but it includes a clear login action so existing users are not trapped
- the tournament header now becomes a finishing-stage panel: third place appears after the third-place match, and a full 1-2-3 podium appears after the final

## Auth follow-up from 2026-04-25

- the Authenticator/TOTP choice now appears before the Google button and email/password form, so the user understands it applies to both signup paths
- Google OAuth started from `/login` now redirects first-time app users back to `/signup` with a clear message instead of silently continuing as a new registration
- the Authenticator setup flow now returns to onboarding after QR verification, preserving the required nickname/profile setup
- refreshes during Authenticator setup now create a fresh QR by clearing stale unverified TOTP factors first, and onboarding redirects back to setup if the MFA choice was not completed
- the background MFA assurance check is visually silent now, so regular reloads no longer flash the Authenticator gate for users who do not need a code
- for MFA-protected accounts, the initial assurance check happens in the server layout, preventing a protected page flash before the code screen
- dashboard live refresh polling now uses a version check instead of refreshing the whole route every 1.5 seconds
- signup/email OTP resend now uses a polished 60-second cooldown timer and no longer exposes provider-specific rate-limit wording
- Authenticator setup now uses Supabase `challengeAndVerify` for the first code, so email/password signups do not fail while starting a separate MFA challenge
- Google OAuth started from `/login` now removes the unused Google-only Auth user when app registration has not started, preventing that ghost account from blocking a later same-email password signup
- stale sessions from users deleted directly in Supabase are now detected by the client AuthProvider and cleared locally, so nav/sidebar controls stop showing an authenticated profile
- `/mfa/setup` now prevents duplicate enrollment races and recovers from `Factor not found` by generating a fresh QR for the current session
- same-email signup for a real existing Google account now uses a real existing-account email OTP and then `updateUser({ password })`, so users can add password login without creating duplicate Supabase users
- same-email password linking no longer holds the chosen password in client state while waiting for OTP; the password is re-entered after code verification before it is linked
- Dev Tools status buttons now write directly to Supabase and trigger the existing tournament sync pipeline immediately, instead of requiring an extra row save for `LIVE`/`FINISH`
- `20260425000022_add_match_phase.sql` adds the DB-backed `match_phase` used for halftime, stoppage time, extra time and penalties across dashboard, tournament and prediction views
- `/mfa/setup` now redirects anyone who did not explicitly choose Authenticator during signup, even if a stale unverified TOTP factor exists, so the enrollment screen cannot be opened casually after registration
- Dev Tools now treats halftime and penalties as minute-less phases, keeps status/phase/minute/save controls visible at the start of the RTL table, and blocks extra time/penalty phases for group-stage matches in both UI and API validation
