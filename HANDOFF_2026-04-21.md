# World Cup Predictions Handoff

Date: 2026-04-21  
Branch target: `master`  
Repo: `https://github.com/MoranTheKing/worldcup-predictions`

## Current State

The tournament engine is now built around the `matches` table as the single source of truth.

Key behaviors now in place:

- Group standings are recalculated from live match data, not from stale cached values on `teams`
- Group tie-break sorting follows the recursive head-to-head rules from `TOURNAMENT_RULES.md`
- Terminal group tables now lock deterministically when no remaining group matches can change the order
- Knockout participants are repopulated through a single idempotent sync path instead of piecemeal updates
- Dev Tools actions now trigger real database sync for the tournament state

## Major Logic Fixes

### 1. Deterministic Position Locking

Implemented in:

- `lib/utils/standings.ts`

What changed:

- Added `isLocked` to `TeamStanding`
- A team now receives `lockedRank` and `isLocked: true` when the group is in a terminal state and the team has completed all three matches
- This fixes the previous issue where a finished team could still look "unlocked" even though no remaining result could move it
- Example case fixed: if two teams finish level on points and one is ahead on goal difference after the last group match, that higher team is immediately locked in place

### 2. Tournament State Sync

Implemented in:

- `lib/tournament/knockout-progression.ts`
- `lib/tournament/tournament-state.ts`

New behavior:

- `syncTournamentState()` now recalculates:
  - all group standings
  - the best 8 third-placed teams
  - all knockout participant slots for matches 73-104
- The sync is idempotent
- Every placeholder-driven knockout slot is re-derived from the current tournament state
- Stale knockout `home_team_id` / `away_team_id` values are cleared automatically when they are no longer valid

Important implementation detail:

- Tournament teams are now built from live `matches` data with zero defaults, rather than falling back to stale cached `teams.points` / `goals_for` / `played_count`
- This fixes the previous bug where `Clear` could still leave standings/bracket data looking "stuck"

## Dev Tools

Updated files:

- `app/dev-tools/DevToolsClient.tsx`
- `app/api/dev/matches/[id]/route.ts`
- `app/api/dev/matches/bulk/route.ts`
- `app/api/dev/matches/clear/route.ts`
- `app/api/dev/matches/randomize/route.ts`

What now works:

- `Save` on a single match updates the DB and runs global tournament sync
- `Save All Matches` bulk-saves the draft state and runs global tournament sync
- `Clear All Match Data` resets scores/statuses and clears/rebuilds knockout participants
- `Randomize All Matches` is now a server-side action:
  - writes random finished results to all matches
  - adds penalties for drawn knockout matches
  - immediately runs global tournament sync
- `Finish All Matches` remains a local drafting helper, followed by `Save All Matches`

## Remaining Caveat

The current project still relies on the seeded knockout placeholder structure already present in `matches_data.json(.txt)`. The sync logic is now stable and reactive, but if product later wants the exact official FIFA Annex C slot table encoded explicitly, `lib/utils/thirdPlaceAllocations.ts` is still the place to finish that full 495-case mapping.

## Bottom Line

The two critical issues that were reported are now addressed:

- group positions no longer remain visually unresolved after a terminal group state
- the knockout bracket no longer depends on stale persisted team IDs and is rebuilt from fresh standings after every relevant dev action
