## Deterministic Group Locking

Implemented in:

- `lib/utils/standings.ts`

The app now locks group positions through three paths:

1. Terminal lock
   The group has no remaining `scheduled` or `live` matches and the team has `played === 3`

2. Scenario lock
   The team has `played === 3` and scenario analysis converges to one exact finishing rank

3. All-played lock
   Every team in the group has `played === 3`, even if some match statuses are still `live`

This also drives the Best 3rd Place terminal-state bypass:

- once all 12 groups are terminal, ranks `1-8` are hard-qualified and ranks `9-12` are hard-eliminated

## Annex C Injection Fix

Implemented in:

- `lib/utils/thirdPlaceAllocations.ts`
- `lib/tournament/knockout-progression.ts`

What changed:

- `calculateThirdPlaceMatchups()` is now side-aware
- It iterates through the Annex C mapping entries, not through placeholders that assume the winner group is always on the home side
- For each mapping like `"A": "H"`, it scans matches `73-88`, finds the match containing `1A`, and assigns the Group H 3rd-place team to the opposite side
- `syncTournamentState()` now writes that assignment back into `home_team_id` or `away_team_id` based on the discovered side

Why this mattered:

- The previous implementation only wrote Annex C results into the `away` side assignment map
- That broke database persistence whenever the winner-group placeholder orientation did not match the assumption baked into the sync path

Trigger condition:

- The DB update runs once the Best 3rd Place table reaches its mathematically terminal locked state

## Knockout Cascade

Main files:

- `lib/tournament/knockout-progression.ts`
- `lib/tournament/knockout-utils.ts`
- `lib/bracket/knockout.ts`
- `lib/tournament/knockout-tree.ts`

Current progression behavior:

- Round of 32 sides resolve from explicit DB team IDs first
- If no explicit team is persisted yet, the app falls back to seeded Round of 32 assignments
- Later knockout rounds resolve recursively from `Winner Match N` and `Loser Match N` placeholders
- `determineKnockoutWinnerId()` and `determineKnockoutLoserId()` handle promotion into later rounds
- The Tournament UI now renders the bracket from the actual winner-tree graph in a vertical mobile-first layout
- Rounds are stacked top-to-bottom and grouped by the exact downstream match they feed into
- Visual connector branches now converge downward from each child pair into the next match, replacing the old helper text approach
- Stage headers stay sticky while the user scrolls the bracket
- The Final and 3rd Place matches now sit together in the bottom block, with gold and bronze visual separation respectively

## Global Elimination Sync

Implemented in:

- `lib/tournament/elimination.ts`
- `lib/tournament/knockout-progression.ts`
- `lib/utils/standings.ts`

Current elimination behavior:

- 4th-place teams are persisted into `teams.is_eliminated` once they are mathematically out
- The bottom 4 teams in the locked Best 3rd Place table are also persisted as eliminated
- Any team that loses a finished knockout match is marked eliminated
- Semi-final losers are the exception: they remain alive until the 3rd-place match is completed, and only the loser of that final classification match is eliminated

Operational effect:

- `syncTournamentState()` now updates both `matches` and `teams`
- Dev Tools single save, bulk save, clear, and randomize all hit the same elimination sync
- Dev Tools `Finish All Matches` now persists immediately through the bulk save path
- Dev Tools per-match `RESET` now persists immediately and forces the score back to `0:0`
- `teamsRemaining` on the Tournament page now comes from the persisted `teams.is_eliminated` flags, so the counter tracks knockout exits correctly
