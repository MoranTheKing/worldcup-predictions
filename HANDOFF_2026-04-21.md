## Important follow-up added on 2026-04-23

Separate from the audit, the auth funnel was also upgraded on 2026-04-23:

- new users now complete a required onboarding step before entering the protected app
- public nicknames are now unique by design, with DB enforcement in `20260423000020_enforce_unique_profile_handles.sql`

Additional follow-up landed later the same day:

- localhost-only dev gating no longer trusts spoofable `X-Forwarded-Host` values
- the predictions hub outright form was visually tightened with less redundant copy
- onboarding now always starts at the profile step before any tournament picks, even for prefilled Google users
- players can now upload a private personal avatar during onboarding and edit nickname/avatar later from the in-app profile card
- uploaded personal avatars are validated server-side, stored in a private bucket, and served through an authenticated internal route instead of a public asset URL
- arbitrary same-origin avatar paths are no longer allowed, so stored avatar URLs cannot trigger authenticated GET requests like `/auth/callback?...` in other users' browsers
- private uploaded-avatar URLs must remain relative `/api/profile/avatar/...` routes; external absolute URLs that mimic the private route are rejected
- after kickoff, new users still go through the profile onboarding step, but locked tournament winner/top-scorer picks are skipped
- avatar uploads now render a live local preview before save, with horizontal/vertical framing controls and zoom
- private avatar crop metadata is persisted without adding new public URL surface area
- real profile avatars across the app can now be clicked to open a larger dismissible preview
- avatar panning now uses the true zoom/crop bounds instead of a static offset, so vertical and horizontal movement stay available across the full reachable frame
- the profile editor now opens from the top with nickname controls before the avatar framing section
- onboarding profile layout now also starts from the top with nickname controls before avatar controls

## Follow-up added on 2026-04-25

- Google OAuth launched from `/login` is treated as login only; first-time Google users are signed out and redirected to `/signup` with a clear registration message

- the global MFA check no longer flashes the Authenticator screen while it is only checking in the background
- root layout now resolves the first MFA gate state server-side, so protected content does not flash before a required Authenticator challenge
- dev live refresh now polls a lightweight match version endpoint and refreshes only on actual match changes
- email OTP cooldown copy is now user-facing only, with a live 60-second countdown before another code can be requested for the same address
- Authenticator setup now verifies the first TOTP code with Supabase `challengeAndVerify`, avoiding the split challenge/verify failure seen after email/password signup
- first-time Google login attempts that have not started app registration now clean up the temporary Google-only Auth user before returning to `/signup`, so the same email can still register with password + OTP
